# Phase 3 — UI Design Contract

> Visual and interaction contract for the frontend shell and smart dashboard. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none (hand-rolled primitives with Tailwind utilities) |
| Icon library | lucide-react ^1.8.0 |
| Font | System font stack (Tailwind default: ui-sans-serif, system-ui, sans-serif) |

**Rationale:** Phase 1 established hand-rolled Button, Input, Toast components using Tailwind utility classes. Phase 3 extends this with Card, Badge, Modal, Sidebar, StatCard, Skeleton. No shadcn — the existing component pattern is lightweight and sufficient.

---

## Spacing Scale

Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px (1) | Icon gaps, inline element padding, gap between icon and label in sidebar |
| sm | 8px (2) | Compact element spacing, badge padding, stat card internal gaps |
| md | 16px (4) | Default element spacing, card padding, sidebar item padding |
| lg | 24px (6) | Section padding within dashboard, content area padding on mobile |
| xl | 32px (8) | Layout gaps between dashboard sections, content area padding on desktop |
| 2xl | 48px (12) | Touch target minimum (mobile sidebar items, buttons) |
| 3xl | 64px (16) | Not used in Phase 3 |

Exceptions: Sidebar width uses 256px (64 * 4) expanded, 64px collapsed (icon-only mode). Touch targets on mobile use minimum 48px (12 * 4) for tappable sidebar items and buttons.

**Source:** Derived from existing components — Button h-8/h-10/h-12, Input px-3/py-2, AuthLayout p-6/p-8, card border rounded-xl with p-6.

## Typography

| Role | Size | Weight | Line Height | Tailwind Class |
|------|------|--------|-------------|----------------|
| Badge | 12px | 600 (semibold) | 1.3 (16px) | text-xs font-semibold |
| Body | 14px | 400 (normal) | 1.5 (20px) | text-sm leading-5 |
| Label | 14px | 600 (semibold) | 1.4 (20px) | text-sm font-semibold |
| Heading | 20px | 600 (semibold) | 1.2 (24px) | text-xl font-semibold |
| Display | 28px | 600 (semibold) | 1.2 (34px) | text-2xl font-semibold |

**Weights used:** 400 (normal) for body text, 600 (semibold) for labels, headings, display values, and badges. Two weights only.

**Usage in Phase 3:**
- **Badge (12px/600):** Status badges in stale apps list, dashboard card badges with count
- **Body (14px/400):** Dashboard list items (stale apps, upcoming interviews, recent activity), sidebar nav labels, modal body text
- **Label (14px/600):** Stat card labels, form field labels (existing Input pattern), sidebar active item
- **Heading (20px/600):** Section headings ("Stale Applications", "Upcoming Interviews", "Recent Activity"), modal title
- **Display (28px/600):** Stat card values (e.g., "12", "3", "8.5%", "14 days"), page title "Dashboard"

**Source:** Existing components use text-sm (14px), text-2xl for page headings, font-medium for labels, font-bold for brand. Collapsed to 2 weights (400/600) for consistency.

## Visual Hierarchy

**Primary visual anchor:** The 4-stat-card grid at the top of the dashboard. This is the first element the eye lands on after the sidebar/header chrome. Stat card values use the largest type size (28px Display) and sit inside elevated Card containers with shadow-sm, creating the strongest visual weight on the page.

**Secondary anchors:** The "Needs Attention" and "Upcoming Interviews" two-column section, followed by the full-width "Recent Activity" feed.

**Accessibility — collapsed sidebar icon-only mode:** When the sidebar collapses to 64px (tablet breakpoint or manual toggle), each nav icon receives both a `title` attribute and an `aria-label` attribute matching the full nav label text (e.g., `title="Dashboard" aria-label="Dashboard"`). This ensures screen readers announce the destination and sighted users see a native tooltip on hover.

## Dashboard Layout

### Section 1: Summary Stat Cards (VIEW-01, D-05)
- Grid: 4 columns on desktop (lg+), 2 columns on tablet (md), 1 column on mobile (<md)
- Gap: 16px (gap-4)
- Cards:
  1. **Total Active** — count of non-rejected/withdrawn/accepted apps. Icon: Briefcase. Trend: vs last 7 days.
  2. **Interviews This Week** — count of apps in "interviewing" status (placeholder until Phase 5 has real interview data). Icon: MessageSquare. Trend: vs last week.
  3. **Offer Rate** — (offer + accepted) / total * 100, rounded to 1 decimal. Icon: TrendingUp. Trend: vs all-time.
  4. **Avg Days in Pipeline** — average days from created to current date for active apps. Icon: Clock. Trend: vs last month.

### Section 2: Two-Column Split (VIEW-02, D-06)
- Grid: 2 columns on desktop (lg+), stacked on mobile
- Gap: 16px (gap-4)
- **Stale Applications** (left column):
  - Card container, heading "Needs Attention", badge with count
  - List: max 5 items, sorted by staleness (most stale first)
  - Each item: company name (font-semibold), role (text-secondary), status badge, "X days ago" timestamp
  - Warning icon (AlertTriangle, amber-500) per item
  - "View all" link at bottom if more than 5
- **Upcoming Interviews** (right column):
  - Card container, heading "Upcoming Interviews"
  - Empty state until Phase 5 builds interview tracking
  - Each item (future): company name, role, date/time, round type badge
  - Calendar icon (Calendar, blue-500) per item

### Section 3: Recent Activity (D-07)
- Full-width Card container, heading "Recent Activity"
- List: max 10 items, sorted by `occurred_at` desc (requires cross-app timeline endpoint)
- Each item: status dot (color from STATUS_COLORS), description text, relative timestamp ("2 hours ago"), company name
- "View all" link at bottom

### Quick-Add (VIEW-03, D-08, D-09)
- Trigger: "+" button in header (primary Button, label "+ Add Application")
- Opens: Modal with title "Add Application"
- Required fields: Company name (Input), Role title (Input), Status (select dropdown, default "wishlist")
- Optional fields section: collapsed by default, "More options" expand toggle
- Optional fields: Priority (select), Location type (select), Salary min/max (number inputs), Source (select), Job URL (Input), Applied date (date input), Notes (textarea)
- Submit CTA: "Add Application" (primary Button)
- Cancel: "Discard" (outline Button) or backdrop click/Escape
- Success: close modal, invalidate applications query (TanStack Query), show success toast
- Error: show error message inline below form, keep modal open

## Interaction States

### Loading States
- **Initial page load:** Full-screen centered spinner (existing pattern from dashboard.tsx) with "Loading..." text
- **Dashboard sections loading:** Skeleton placeholders matching final layout. StatCard skeleton: 4 pulsing rectangles in grid. List sections: 3 skeleton rows each.
- **Quick-add form submitting:** Submit button enters `loading` state (existing Button loading prop with spinner)

### Error States
- **API fetch failure:** Error card replaces section content with AlertCircle icon (red-500), heading, body copy, and "Try again" button
- **Quick-add validation:** Inline red error text below each field (existing Input error pattern)
- **Quick-add API failure:** Toast notification with error message, modal stays open

### Empty States
- **No applications at all:** Full dashboard replaced with centered illustration area (no image — just icon + text), heading, body, and prominent CTA button
- **No stale apps:** Green checkmark icon + "All caught up" message inside the card
- **No upcoming interviews:** Calendar icon + "No interviews scheduled" message
- **No recent activity:** Clock icon + "No activity yet" message
- **Placeholder pages:** Centered content with page icon, heading, body, and "Go to Dashboard" link

### Hover States
- **Sidebar nav items:** bg-stone-100 / bg-zinc-800, 150ms transition
- **Cards in lists (stale apps, activity):** Subtle bg-stone-50 / bg-zinc-800/50 highlight on row hover
- **Buttons:** Follow existing Button component hover states
- **Quick-add trigger:** Standard primary button hover (bg-stone-700)

### Focus States
- **All interactive elements:** ring-2 ring-stone-400 / ring-zinc-400, ring-offset-2 (existing pattern from Button)
- **Sidebar nav items:** Same ring treatment on keyboard focus
- **Modal focus trap:** Focus cycles within modal when open

### Responsive Breakpoints
- **< 768px (mobile):** Sidebar hidden, hamburger toggle in header, stat cards 1-col, dashboard sections stacked, content padding 16px
- **768px-1023px (tablet):** Sidebar collapsed to icon-only (64px), stat cards 2-col, dashboard sections stacked, content padding 24px
- **>= 1024px (desktop):** Sidebar expanded (256px), stat cards 4-col, stale/interviews 2-col, content padding 32px

## Dark Mode Contract (UI-01, D-14 through D-17)

| Aspect | Specification |
|--------|--------------|
| Strategy | Class-based via `@custom-variant dark (&:where(.dark, .dark *));` in index.css |
| Toggle mechanism | Zustand store with persist middleware, key "jobdash-theme" in localStorage |
| Modes | "system" (default on first visit), "light", "dark" |
| System detection | `window.matchMedia("(prefers-color-scheme: dark)")` with change listener |
| Application | `document.documentElement.classList.toggle("dark", isDark)` |
| Transition | No global transition (prevents FOUC). Individual elements transition colors via component-level `transition-colors duration-150`. |
| All components | Must use semantic Tailwind classes with `dark:` variant. Never hardcode light-only colors. |
